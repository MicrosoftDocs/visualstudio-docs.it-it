# <a name="ca1508-avoid-dead-conditional-code"></a>CA1508: Evitare codice di condizione non utilizzato

|Elemento|valore|
|-|-|
|CheckId|Ca1508|
|Category|Microsoft. gestibilità|
|Modifica|Senza interruzioni|

## <a name="cause"></a>Causa

Un metodo ha codice condizionale che restituisce sempre `true` o `false` in fase di esecuzione. In questo modo viene creato codice non attivo nel `false` ramo della condizione.

## <a name="rule-description"></a>Descrizione della regola

I metodi possono avere codice condizionale, ad esempio istruzioni If, espressioni binarie ( `==` , `!=` , `<` , `>` ), controlli null e così via. Si consideri, ad esempio, il codice seguente:

```csharp
public void M(int i, int j)
{
    if (i != 0)
    {
        return;
    }

    if (j != 0)
    {
        return;
    }

    // Below condition will always evaluate to 'false' as 'i' and 'j' are both '0' here.
    if (i != j)
    {
        // Code in this 'if' branch is dead code.
        // It can either be removed or refactored.
        ...
    }
}
```

I compilatori C# e VB eseguono l'analisi dei controlli condizionali che coinvolgono _valori costanti_ in fase di compilazione che restituiscono sempre `true` o `false` . Questo analizzatore esegue l'analisi del flusso di dati di variabili non costanti per determinare controlli condizionali ridondanti che coinvolgono _valori non costanti_. Nel codice precedente, l'analizzatore determina che `i` e `j` sono entrambi `0` per tutti i percorsi del codice che raggiungono il `i != j` controllo. Di conseguenza, questo controllo restituirà sempre `false` in fase di esecuzione. Il codice contenuto nell'istruzione If è codice inattivo e può essere rimosso o sottoposto a refactoring. Analogamente, l'analizzatore rileva i valori Null delle variabili e segnala i controlli null ridondanti.

> [!NOTE]
> Questo analizzatore esegue un'analisi costosa del flusso di flussi di valori non costanti. Questo può aumentare il tempo complessivo di compilazione su determinate basi di codice.

## <a name="when-to-suppress-warnings"></a>Quando escludere gli avvisi

È possibile eliminare una violazione di questa regola se non si è preoccupati della gestibilità del codice. È anche possibile escludere le violazioni identificate come falsi positivi. Queste sono possibili in presenza di codice simultaneo che può essere eseguito da più thread.
